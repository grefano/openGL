Entendido. Então o reader só **abre/gerencia o arquivo**, e o Renderer é quem sabe o que fazer com ele.

Nesse caso a resposta para o seu dilema é: **não crie `FileReader` abstrato agora**. O `MasterClip` pode usar `std::variant` ou simplesmente um `void*` tipado — mas a solução mais limpa é o próprio `MasterClip` ser polimórfico:

```cpp
// MediaSource já é polimórfico — você disse isso
struct MediaSource {
    std::filesystem::path filepath;
};
struct VideoSource : MediaSource { /* fps, duration, codec info... */ };
struct ImageSource : MediaSource { /* width, height, color space... */ };

// MasterClip também é polimórfico — sem FileReader abstrato forçado
class MasterClip {
public:
    virtual ~MasterClip() = default;
    MediaSource* source;
};

class VideoMasterClip : public MasterClip {
public:
    VideoReader reader; // concreto, sem interface forçada
};

class ImageMasterClip : public MasterClip {
public:
    ImageReader reader;
};
```

O Renderer então faz o `dynamic_cast` — ou melhor, o **visitor**:

```cpp
class Renderer {
public:
    Frame render(const TimelineClip& clip, Timecode tc) {
        return dispatch(*clip.masterClip, tc);
    }

private:
    Frame dispatch(const VideoMasterClip& c, Timecode tc) {
        auto data = c.reader.read(tc); // retorna dados brutos
        return processVideo(data);
    }
    Frame dispatch(const ImageMasterClip& c, Timecode tc) {
        auto data = c.reader.read();
        return processImage(data);
    }
};
```

---

O ponto central: **`FileReader` abstrato só vale se o Renderer for tratar todos os readers igual** — o que você não quer. Polimorfismo no `MasterClip` resolve isso sem interface vazia.
